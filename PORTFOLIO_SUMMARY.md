# KoK 포트폴리오 정리

## 프로젝트 개요

KoK는 여러 사람이 각자의 출발 위치를 입력하면, 모두가 납득할 수 있는 약속 후보지를 추천하고 게임형 랜덤 추첨으로 최종 장소를 정하는 지도 기반 약속 장소 결정 앱입니다.

기존의 약속 장소 결정은 대화가 길어지고, 결국 익숙한 지역으로만 수렴하는 문제가 있었습니다. 이 앱은 위치 입력, 후보 추천, 공정성 검토, 랜덤 추첨, 주변 장소 탐색까지 한 흐름으로 연결해 장소를 정하는 과정을 더 빠르고 재미있게 만드는 것을 목표로 했습니다.

## 핵심 문제 정의

- 여러 명이 흩어져 있을 때 중간 장소를 정하는 과정이 번거롭다.
- 단순 중간 지점만 계산하면 실제 이동 시간이나 동선의 납득도가 떨어질 수 있다.
- 후보가 너무 뻔하거나 특정 지역에 치우치면 랜덤 경험의 재미가 줄어든다.
- 온라인으로 함께 정할 때는 참여자 동기화, 방 권한, 다시뽑기 기준이 필요하다.

## 주요 사용자 흐름

1. 홈 화면에서 게스트로 시작하거나 로그인 후 온라인 약속방을 만든다.
2. 참여자는 주소 검색, 현재 위치, 지도 선택, 저장된 친구/프로필 위치를 통해 출발지를 입력한다.
3. 모임 카테고리, 선정 방식, 랜덤 강도, 교통수단을 선택한다.
4. 앱은 기본 후보 데이터, AI 후보 선정, 지도 검색 결과, 이동거리/이동시간 기준을 조합해 후보지를 만든다.
5. 후보는 지도와 카드 형태로 확인하고, 원하지 않는 후보는 제외할 수 있다.
6. 카드 셔플, 사다리타기, 돌림판 같은 게임형 추첨으로 최종 장소를 정한다.
7. 결과 화면에서 참여자별 경로, 주변 맛집/카페/놀거리, 다시뽑기 흐름을 확인한다.

## 앱 구조

### Frontend

- React, TypeScript, Vite 기반의 모바일 중심 웹 앱
- `HomeScreen`, `PlannerScreen`, `RandomDrawer`, `ResultScreen` 중심의 화면 구조
- 지도, 후보 카드, 참여자 입력, 온라인 방 상태, 결과 확인을 기능별 컴포넌트로 분리
- Tailwind 기반 스타일과 Radix UI 계열 컴포넌트를 활용해 빠르게 인터랙션 구성

### 장소 후보 생성

- 기본 후보 데이터와 참여자 위치를 바탕으로 후보 유니버스를 생성
- 모임 카테고리별 후보 적합도, 중심점 거리, 참여자별 이동 편차, 최대 이동시간을 함께 계산
- 가까운 참여자끼리는 검색 반경을 줄이고, 너무 멀리 벗어난 후보를 제외
- 밸런스 모드, 핫플 모드, 집앞 모드에 따라 후보 선정 기준을 다르게 적용

### AI 및 외부 API

- OpenAI 또는 Upstage API를 활용해 후보 순서를 보정하고 다양한 상권 후보를 구성
- AI 결과는 바로 사용하지 않고, 허용된 후보 ID와 fallback 후보 로직을 통해 안정성을 확보
- 네이버 지도/지역 검색으로 주소 검색, 지도 표시, 주변 장소 탐색을 처리
- ODsay 대중교통 API와 네이버 길찾기 API를 활용해 실제 경로 정보를 보강

### 데이터 및 온라인 방

- Supabase Auth로 로그인과 프로필을 관리
- `profiles`, `saved_friends`, `meeting_rooms`, `meeting_room_participants` 테이블로 사용자, 친구, 방, 참여자 데이터를 분리
- 온라인 방에서는 참여자 추가, 레디 상태, 결과 공유, 과반 동의 기반 다시뽑기 흐름을 지원
- 실시간 구독과 주기적 동기화를 함께 사용해 방 상태가 끊기지 않도록 처리

## 핵심 기능

- 참여자별 출발 위치 입력 및 저장 친구 빠른 추가
- 지도 기반 후보 표시와 참여자 위치 시각화
- 카테고리별 약속 후보 추천
- 이동시간 편차를 고려한 공정성 기반 후보 필터링
- AI 기반 후보 다양화 및 heuristic fallback
- 대중교통/자차 경로 확인
- 온라인 약속방 생성, 참여, 레디, 다시뽑기 투표
- 카드 셔플, 사다리타기, 돌림판 기반 랜덤 추첨
- 결과 장소 주변의 맛집, 카페, 활동 추천

## 후보 선정 로직

KoK의 후보 선정은 단순히 중간 좌표를 하나 계산하는 방식이 아니라, 후보 생성, 이동 부담 계산, 모드별 점수화, AI 보정, 실제 경로 검증, 추첨 풀 압축 순서로 진행됩니다.

### 1. 후보 유니버스 생성

먼저 참여자 위치와 선택한 카테고리를 바탕으로 후보가 될 수 있는 전체 목록을 만듭니다. 이 목록에는 미리 준비된 주요 상권 데이터뿐 아니라, 참여자들의 중간 지점, 가까운 참여자 그룹을 위한 좁은 반경 후보, 각 참여자 근처 후보, 랜덤 강도에 따라 추가되는 로컬 후보가 함께 들어갑니다.

이후 이름이 같거나 좌표가 지나치게 가까운 후보는 중복으로 보고 정리합니다. 다만 집앞 모드처럼 일부러 각 참여자 근처 후보를 섞어야 하는 경우에는 로컬 후보가 사라지지 않도록 예외를 둡니다.

### 2. 후보별 이동 부담 계산

각 후보마다 모든 참여자의 예상 이동 정보를 계산합니다. 참여자의 교통수단이 대중교통이면 대중교통 기준 예상 시간과 비용을, 자차이면 자동차 기준 예상 시간과 유류비를 계산합니다.

후보별로 보는 주요 지표는 다음과 같습니다.

- 평균 이동시간
- 가장 오래 걸리는 참여자의 이동시간
- 가장 가까운 참여자와 가장 먼 참여자의 이동시간 차이
- 모든 참여자가 각자의 최대 이동시간 안에 도착 가능한지 여부
- 후보가 참여자들의 중심점에서 얼마나 떨어져 있는지
- 두 명 이상일 때 참여자 사이의 중심축에서 얼마나 벗어났는지
- 선택한 모임 카테고리와 후보의 성격이 맞는지

이 단계에서 중요한 기준은 "완벽한 중앙"이 아니라 "모두가 납득할 수 있는 이동 부담"입니다.

### 3. 기본 점수화

기본 정렬에서는 모든 참여자가 도착 가능한 후보를 우선합니다. 그다음 이동시간 편차, 평균 이동시간, 가장 긴 이동시간, 중심점 거리, 중심축 이탈 정도, 카테고리 적합도를 함께 반영합니다.

특히 한 사람만 지나치게 가깝고 다른 사람이 멀어지는 후보에는 페널티를 줍니다. 두 명이 만나는 경우에는 두 출발지를 잇는 동선에서 너무 옆으로 벗어난 후보도 낮게 평가합니다.

### 4. 모드별 후보 선정

밸런스 모드는 공정성을 가장 강하게 봅니다. 랜덤 강도에 따라 허용되는 이동시간 편차를 다르게 두고, 낮은 단계에서는 더 균형 잡힌 후보만 남깁니다. 참여자들이 가까운 거리에 모여 있으면 검색 범위를 자동으로 줄여 먼 유명 상권이 끼어들지 않게 합니다.

핫플 모드는 공정성보다 상권 매력과 카테고리 적합도를 더 크게 봅니다. 그래도 이동 부담이 너무 큰 후보는 밀어내고, 도착 가능한 후보 안에서 식사, 카페, 술자리, 문화 활동 등 선택한 목적에 맞는 상권을 우선합니다.

집앞 모드는 일부러 누군가의 생활권 근처 후보를 섞는 모드입니다. 랜덤 강도가 높을수록 참여자별 로컬 후보가 더 적극적으로 들어가며, 최고 단계에서는 각 참여자 근처 후보가 최소한 한 번씩 후보군에 포함되도록 보정합니다.

### 5. AI 후보 보정

AI는 전체 후보를 새로 만들어서 결정하는 역할이 아니라, 이미 계산된 후보 목록 안에서 더 자연스러운 순서와 조합을 고르는 보조 레이어로 사용됩니다.

앱은 후보 ID, 이동시간 지표, 카테고리, 선정 모드, 랜덤 강도를 API에 전달합니다. OpenAI 또는 Upstage가 연결되어 있으면 AI가 허용된 후보 ID 안에서 추천 순서를 반환합니다. API 키가 없거나 호출이 실패하면 미리 계산한 heuristic 후보 목록을 그대로 사용합니다.

이 구조 덕분에 AI 결과가 불안정하더라도 앱은 계속 작동하고, AI가 넓은 지역명이나 엉뚱한 장소를 임의로 끼워 넣는 문제도 줄일 수 있습니다.

### 6. 실제 경로 기반 재검증

밸런스 모드에서는 AI 또는 기본 로직으로 추린 후보를 다시 실제 경로 정보로 검증합니다. 대중교통은 ODsay, 자차는 네이버 길찾기 데이터를 활용하고, 외부 API가 실패하면 예상 이동시간으로 fallback합니다.

이 단계에서는 이동시간 편차가 낮아 보여도 모두가 오래 이동해야 하는 후보, 중심축에서 많이 벗어난 후보, 실제 경로 기준으로 부담이 커진 후보를 다시 걸러냅니다.

### 7. 최종 추첨 풀 생성

마지막으로 `getDrawPool`에서 실제 랜덤 추첨에 들어갈 후보 풀을 만듭니다. 밸런스 모드에서는 실제 경로까지 검증된 공정 후보를 우선하고, 후보가 너무 적으면 기준을 단계적으로 완화합니다. 그래도 중심축에서 크게 벗어나거나 한쪽에 치우친 후보는 제외합니다.

핫플 모드와 집앞 모드는 각각 상권 매력, 로컬 후보 포함 비율에 맞게 추첨 풀을 구성합니다. 사용자가 후보를 직접 제외한 경우에는 해당 후보를 풀에서 제거합니다.

최종 선택은 후보 풀 안에서 가중 랜덤으로 이루어집니다. 완전 동일 확률이 아니라 카테고리 적합도, 접근성, 공정성, 랜덤 강도, 후보 분위기를 가중치로 반영해 "랜덤이지만 납득 가능한 결과"가 나오도록 설계했습니다.

### 후보 선정 흐름 요약

```text
참여자 위치 입력
→ 기본 상권 + 중간 지점 + 로컬 후보 생성
→ 후보별 이동시간, 편차, 중심점 거리 계산
→ 밸런스/핫플/집앞 모드별 1차 점수화
→ AI가 허용된 후보 안에서 순서 보정
→ 실제 대중교통/자차 경로로 공정성 재검증
→ 사용자가 제외한 후보 제거
→ 최종 추첨 풀 생성
→ 가중 랜덤으로 최종 장소 선택
```

## 시행착오와 개선 과정

### 1. 비용 계산 중심에서 결정 경험 중심으로 전환

초기에는 교통비 차이까지 고려한 공정한 약속 장소 계산기를 구상했습니다. 하지만 사용자가 가장 크게 느끼는 가치는 비용 계산보다 "어디서 볼지 빠르게 정하는 것"과 "랜덤으로 고르는 재미"에 있었습니다.

그래서 MVP에서는 비용 계산 기능을 제외하고, 장소 후보 추천과 추첨 경험을 중심으로 방향을 바꿨습니다. 이를 통해 서비스의 목적이 더 명확해졌고, 화면 흐름도 단순해졌습니다.

### 2. 지도 상태 동기화 문제

후보가 바뀔 때마다 지도를 강제로 이동시키면 사용자가 직접 보고 있던 화면이 튀는 문제가 있었습니다. 반대로 지도 갱신을 너무 적게 하면 새 후보가 반영되지 않는 것처럼 보였습니다.

이를 해결하기 위해 후보 변경 시점에는 전체 참여자와 후보가 보이도록 bounds를 맞추고, 사용자가 직접 조작 중일 때는 강제 이동을 최소화하는 방식으로 정리했습니다.

### 3. 주소 검색 정확도 문제

사용자가 "역삼역"처럼 장소명을 입력했을 때 도로명 주소나 유사 키워드가 먼저 나오는 경우가 있었습니다. 검색 API는 넓은 결과를 반환하기 때문에, 입력어와 장소명의 일치도, 역/상권 키워드, 실제 장소명 유무를 기준으로 우선순위를 조정했습니다.

이 과정에서 주소 검색은 단순히 결과를 많이 보여주는 기능이 아니라 사용자의 의도를 잘 맞히는 기능이어야 한다는 점을 확인했습니다.

### 4. AI 후보 생성의 제약 설계

초기 AI 프롬프트는 넓은 지역명이나 유명 상권으로 결과가 반복되는 문제가 있었습니다. 예를 들어 "수원"처럼 넓은 지역명이 아니라 "수원역", "행궁동", "인계동"처럼 실제 약속을 잡을 수 있는 상권 단위가 필요했습니다.

이후 AI에게 후보 형식, 검색 가능성, 지역 단위, 참여자별 로컬 후보 포함 조건을 명확히 주면서 후보 품질이 개선되었습니다. AI는 자동 결정자가 아니라 후보 다양화를 돕는 보조 레이어로 정의했습니다.

### 5. 공정성 기준 재정의

처음에는 중심점과의 거리만으로 후보를 평가했지만, 실제 사용자가 느끼는 공정성은 이동시간 차이와 최대 이동 부담에 더 가까웠습니다.

따라서 후보별 평균 이동시간, 가장 오래 걸리는 참여자, 이동시간 편차, 중심축 이탈 정도를 함께 계산했습니다. 특히 가까운 참여자 그룹에서는 먼 후보가 섞이지 않도록 별도의 좁은 반경 기준을 적용했습니다.

### 6. 실시간 방 상태 안정화

온라인 방에서는 참여자 추가, 레디, 결과 확정, 다시뽑기 투표가 동시에 일어나기 때문에 로컬 상태와 DB 상태가 쉽게 어긋났습니다.

이를 줄이기 위해 방 정보와 참여자 목록을 분리하고, 실시간 구독과 polling을 함께 사용했습니다. 또한 방장 권한, 참여자 actor ID, 레디 완료 조건, 과반 다시뽑기 조건을 명시적으로 계산하도록 구조를 정리했습니다.

### 7. UX 복잡도 줄이기

기능이 늘어나면서 후보 설명, AI 상태, 옵션, 주변 추천, 경로 정보가 한 화면에 몰리는 문제가 있었습니다.

최종적으로는 사람 추가, 옵션 선택, 후보 확인, 추첨, 결과 확인 순서로 흐름을 단순화했습니다. 고급 옵션과 후보 목록은 필요할 때 열 수 있게 만들고, 핵심 행동 버튼이 먼저 보이도록 정리했습니다.

## 기술적으로 신경 쓴 점

- AI API가 없어도 heuristic 후보 선정으로 앱이 작동하도록 fallback 구조를 설계
- 후보 추천에서 거리, 이동시간, 편차, 중심축 이탈, 카테고리 적합도를 함께 계산
- 대중교통/자차 경로 API가 실패해도 예상 경로 정보로 자연스럽게 대체
- 온라인 방에서 참여자 ID, 로그인 사용자 ID, 로컬 참여자 ID를 구분해 권한 계산
- 레디 상태와 다시뽑기 투표를 DB에 저장해 여러 사용자가 같은 결과를 공유
- 모바일에서 빠르게 사용할 수 있도록 입력 화면과 후보 확인 화면의 밀도를 조절

## 프로젝트를 통해 배운 점

- 좋은 랜덤은 완전한 무작위가 아니라, 사용자가 납득할 수 있는 범위 안에서 의외성을 주는 것이다.
- 지도 기반 UI는 일반 화면보다 상태 동기화와 사용자 조작 흐름을 더 조심스럽게 다뤄야 한다.
- AI 기능은 명확한 제약과 검증 로직이 있을 때 제품 기능으로 안정화된다.
- 실시간 협업 기능은 나중에 붙이는 부가 기능이 아니라 초기 상태 구조부터 고려해야 한다.
- 기능을 많이 보여주는 것보다 사용자가 지금 해야 할 행동을 명확하게 만드는 것이 더 중요하다.

## 앞으로의 개선 방향

### 1. 실제 이동시간 기반 추천 고도화

현재는 거리 기반 계산과 실시간 경로 정보를 함께 사용하지만, 후보 선정 단계에서 모든 후보의 실제 대중교통/자차 이동시간을 더 적극적으로 반영하면 추천 신뢰도를 높일 수 있습니다.

### 2. 후보 추천 이유 시각화

사용자가 "왜 이 후보가 나왔는지" 더 쉽게 이해할 수 있도록 평균 이동시간, 편차, 가장 먼 참여자, 카테고리 적합도를 간단한 시각 요소로 보여줄 수 있습니다.

### 3. 온라인 방 상태 모델 정리

방 생성, 참여자 입력, 레디, 추첨, 결과, 다시뽑기를 명확한 상태 머신으로 정리하면 실시간 동기화 오류를 더 줄일 수 있습니다.

### 4. 랜덤 추첨 경험 확장

카드 셔플, 사다리타기, 돌림판 외에도 지도 위 핀 드롭, 다트형 선택, 컵 선택 등 사용자가 직접 고르는 느낌을 강화하는 추첨 방식을 추가할 수 있습니다.

### 5. 개인화 추천 강화

사용자 프로필의 선호 카테고리, 자주 만나는 친구, 이전 선택 기록을 활용하면 모임 성격에 맞는 후보와 주변 콘텐츠 추천을 더 정교하게 만들 수 있습니다.

### 6. 테스트 및 안정성 보강

주소 검색, 후보 선정, 온라인 방 동기화, 다시뽑기 투표처럼 핵심 흐름에 대한 단위 테스트와 E2E 테스트를 추가하면 서비스 안정성을 높일 수 있습니다.

## 한 줄 정리

KoK는 약속 장소를 단순히 계산하는 앱이 아니라, 여러 사람이 함께 납득할 수 있는 후보를 만들고 랜덤 추첨으로 결정의 부담을 줄이는 지도 기반 약속 장소 결정 앱입니다.
