# Drop 프로젝트 회고록

## 1. 시작점

처음 아이디어는 수도권에서 여러 명이 약속 장소를 정할 때 생기는 불편함에서 출발했다.  
친구들이 서울, 경기, 인천에 흩어져 있으면 "어디서 볼까?"라는 질문 하나로 대화가 길어지고, 결국 늘 가던 강남, 홍대, 사당 같은 익숙한 지역으로 돌아가는 경우가 많았다.

그래서 초기 기획은 단순했다.

- 각자의 출발 위치를 입력한다.
- 모두가 접근 가능한 중간 후보지를 찾는다.
- 그 안에서 랜덤으로 약속 장소를 뽑는다.
- 가능하면 교통비 차이까지 정산한다.

초기 이름은 랜덤밋이었고, 핵심 문장은 "현실적인 범위 안에서의 랜덤"이었다.

## 2. 기획이 바뀐 이유

개발을 시작하면서 가장 먼저 깨달은 것은 이 서비스의 본질이 "정산"이 아니라 "결정의 재미"라는 점이었다.  
교통비 정산은 논리적으로는 좋아 보였지만, 사용자가 처음 들어왔을 때 느끼는 매력과는 거리가 있었다. 오히려 화면이 복잡해지고, 서비스가 약속 장소 뽑기가 아니라 비용 계산 서비스처럼 보일 위험이 있었다.

그래서 방향을 바꿨다.

- 교통비 정산은 MVP 핵심에서 제외한다.
- 장소 결정 자체를 더 짜릿하게 만든다.
- 선정된 지역 주변의 맛집, 카페, 놀거리 추천으로 확장한다.
- 사용자는 복잡한 설정을 이해하는 사람이 아니라, 빠르게 장소를 정하고 싶은 사람으로 본다.

이후 서비스 이름도 더 짧고 감각적인 `Drop`으로 바뀌었다. 장소가 "떨어진다", 후보가 "드롭된다", 운명이 "찍힌다"는 느낌이 랜덤 경험과 더 잘 맞았다.

## 3. 가장 많이 부딪힌 문제들

### 3-1. 지도 연결 문제

초기에는 네이버 지도 API를 붙이는 과정에서 지도가 보이지 않거나, 확대/축소가 어색하거나, 맵이 특정 위치에 고정되는 문제가 반복됐다.  
가장 큰 원인은 지도 자체보다도 지도 상태를 React 상태와 동기화하는 방식이었다.

특히 후보가 바뀔 때마다 지도 중심을 강제로 옮기면 사용자가 직접 보고 있던 지도가 튕기는 느낌이 생겼다. 반대로 너무 적게 갱신하면 새 후보가 나와도 지도는 그대로라 "고정된 것 같다"는 문제가 생겼다.

결국 배운 점은 지도는 일반 UI보다 훨씬 조심스럽게 다뤄야 한다는 것이다.

- 후보가 바뀌었을 때만 자연스럽게 범위를 맞춘다.
- 사용자가 조작 중일 때는 강제 이동을 최소화한다.
- 핀, 후보, 참여자 위치가 동시에 보이도록 지도 bounds를 계속 조정한다.
- 확대/축소, 드래그 같은 기본 지도 경험은 절대 깨지지 않게 한다.

### 3-2. 주소 검색 정확도 문제

주소 검색도 생각보다 단순하지 않았다.  
"역삼역"을 검색했는데 강남역이 나오거나, 사용자가 기대한 장소명이 아니라 도로명 주소만 나오는 문제가 있었다.

이 문제는 검색 API가 사용자의 의도보다 넓은 결과를 반환하기 때문에 생겼다. 사용자는 "역삼역"이라고 치면 역삼역을 원하지만, 검색 결과는 가까운 주소, 유사 키워드, 상권명까지 섞어 보여줄 수 있다.

그래서 우선순위가 필요했다.

- 입력어와 장소명이 정확히 일치하는 결과를 먼저 보여준다.
- 역 이름 검색은 역, 지하철, 장소 결과를 주소보다 우선한다.
- 도로명 주소만 있는 결과보다 실제 장소명이 있는 결과를 우선한다.
- 사용자가 입력한 키워드가 결과 이름에 포함되는지를 강하게 반영한다.

장소 검색은 "많이 찾는 것"보다 "사용자가 방금 의도한 것"을 맞히는 게 더 중요했다.

### 3-3. AI 후보 생성 문제

AI를 붙인 이유는 단순히 멋있어 보이기 위해서가 아니었다.  
기본 후보 데이터만 쓰면 매번 신도림, 사당, 강남 같은 뻔한 결과가 반복됐고, 수도권 전체를 커버하기도 어려웠다.

AI에게 기대한 역할은 다음이었다.

- 참여자 위치를 보고 중간 상권을 다양하게 제안한다.
- 서울뿐 아니라 경기, 인천 후보도 자연스럽게 포함한다.
- 시/구 단위가 아니라 실제 약속을 잡는 동네/상권 단위로 반환한다.
- 집앞 모드에서는 각 참여자의 생활권 근처 후보도 공정하게 포함한다.

하지만 초반 프롬프트는 너무 넓었다.  
"수도권 후보를 추천해줘"라고 하면 AI는 넓은 행정구역명이나 유명한 서울 상권을 반복해서 내놓았다.

그래서 프롬프트를 더 구체화했다.

- `수원` 단독 금지, `수원역`, `행궁동`, `인계동`처럼 반환
- `인천` 단독 금지, `부평 문화의거리`, `구월 로데오`, `송도 센트럴파크`처럼 반환
- 모든 후보는 네이버 지역 검색으로 바로 검색 가능한 이름이어야 함
- 가까운 참여자끼리는 검색 반경을 좁히고 억지 후보를 만들지 않음
- 집앞 모드는 참여자별로 극단 후보를 최소 1개씩 포함

AI는 마법 버튼이 아니라, 좋은 제약을 줬을 때 쓸모 있는 후보 생성기가 된다는 걸 확인했다.

### 3-4. 공정성 문제

가장 자주 나온 문제는 "왜 한 사람에게만 너무 멀어 보이냐"였다.  
중간 후보를 뽑는다고 했지만, 실제 지도에서 보면 어떤 사람 근처 후보는 없고 다른 사람 쪽 후보만 몰리는 경우가 있었다.

원인은 크게 세 가지였다.

- 직선거리 기준만으로는 실제 이동 시간이 반영되지 않았다.
- 후보 데이터가 특정 지역에 몰려 있으면 공정한 계산을 해도 결과가 편향됐다.
- AI 후보와 기본 후보가 섞이는 과정에서 가까운 로컬 후보가 밀려났다.

그래서 후보 선정 기준을 바꿨다.

- 중간 후보는 중심점과 각 참여자 사이의 거리 편차를 본다.
- 가까운 사람들끼리는 후보 반경을 더 좁힌다.
- 너무 억지스러운 먼 후보는 제외한다.
- 집앞 모드는 각 참여자별 로컬 후보를 하나씩 공정하게 넣는다.
- 향후에는 직선거리뿐 아니라 대중교통/자차 소요시간까지 점수에 반영한다.

이 기능의 핵심은 "완벽한 중간"이 아니라 "납득 가능한 랜덤"이다.

### 3-5. DB와 공유방 문제

Supabase를 붙이면서 친구 저장, 회원가입, 방 만들기, 공유방 참여 기능을 확장했다.  
하지만 여기서 가장 많이 터진 부분도 공유방이었다.

특히 두 번째 사람이 방에 들어오거나, 실시간 참여자 추가가 발생할 때 로컬과 Vercel에서 앱이 멈추는 문제가 있었다. 원인은 실시간 구독, 프로필 생성, 방 상태 갱신이 서로 얽히면서 렌더링과 DB 업데이트가 반복되는 구조에 가까웠다.

여기서 배운 점은 실시간 기능은 나중에 붙이는 장식이 아니라 처음부터 상태 구조를 잘 잡아야 하는 기능이라는 것이다.

- 방장은 랜덤 버튼을 누를 수 있다.
- 지역 다시뽑기는 과반수 동의가 있을 때만 가능하다.
- 참여자는 자기 위치를 입력하고 방에 반영된다.
- 실시간 업데이트는 필요한 데이터만 구독한다.
- 프로필 생성과 방 참여 로직은 분리한다.

실시간 공유방은 재미있는 기능이지만, MVP에서는 안정성이 재미보다 먼저다.

### 3-6. UX가 복잡해진 문제

초반 화면은 기능을 많이 보여주려다 보니 너무 설명이 많았다.  
후보 상세 설명, AI 연결 상태, 옵션 카드, 추천 이유, 태그, 후보 목록, 주변 정보가 한 화면에 몰리면서 사용자가 실제로 해야 할 행동이 흐려졌다.

사용자가 계속 말한 방향은 명확했다.

- 심플 이즈 베스트
- 지금 필요한 것만 보여줘
- 옵션은 숨길 수 있게
- 후보는 보고 싶을 때만 열기
- 긴 설명보다 바로 선택 가능한 UI
- 원하지 않는 후보는 `-` 버튼으로 쉽게 제거

그래서 UX 기준을 다시 잡았다.

- 시작은 지도 중심이어야 한다.
- 사람 추가가 먼저다.
- 그 다음 옵션 선택이다.
- 후보 목록은 접고 펼칠 수 있어야 한다.
- 주변 정보는 후보를 확정하거나 후보 상세 안에서 본다.
- 설명 텍스트는 최대한 줄이고 행동 버튼을 강조한다.

결국 Drop은 설명하는 앱이 아니라 "바로 같이 정하는 앱"이어야 한다.

## 4. 랜덤 경험에 대한 시행착오

초기 랜덤 UI는 룰렛에 가까웠다.  
하지만 룰렛은 너무 흔하고, 기다리는 동안 보는 재미가 약했다.

이후 번개가 지도에 떨어지는 연출, 카드 뽑기, 컵 야바위, 다트 던지기, 지하철 추첨 같은 아이디어를 실험했다.  
그 과정에서 중요한 기준이 생겼다.

- 사용자가 게임을 고르는 것이 아니라, 후보를 고르는 행위가 게임이어야 한다.
- 카드 뽑기라면 진짜 카드를 고르는 느낌이어야 한다.
- 야바위라면 컵 안에 이미 후보가 들어 있고, 사용자가 컵을 선택해야 한다.
- 다트라면 지도 위에 던지는 느낌이 있어야 한다.
- 설명은 빼고, 효과와 상호작용으로 이해되게 해야 한다.

즉, 랜덤 경험은 "연출 영상"이 아니라 "내가 직접 뽑았다"는 감각이 중요했다.

## 5. 현재 제품 방향

현재 Drop의 방향은 다음과 같이 정리된다.

- 여러 명이 위치를 입력한다.
- AI와 지도 검색을 활용해 동네/상권 단위 후보를 만든다.
- 중간에서 만날지, 누군가 집앞까지 갈지 선택한다.
- 랜덤 강도에 따라 무난한 후보부터 극단 후보까지 섞는다.
- 후보는 지도에 표시되고, 원하지 않으면 제외할 수 있다.
- 랜덤 추첨은 카드, 컵, 다트, 핀 드롭 등 게임처럼 진행된다.
- 선정된 장소 주변의 맛집, 카페, 놀거리, 예약 검색까지 이어진다.
- 온라인 방에서는 방장과 참여자 권한을 나눠 함께 결정한다.

초기 기획보다 훨씬 명확해졌다.  
이제 서비스는 "공정한 중간 장소 계산기"가 아니라 "친구들과 장소를 재밌게 결정하는 지도 기반 랜덤 약속 앱"에 가깝다.

## 6. 잘한 결정

- 교통비 정산을 과감히 MVP에서 뺀 것
- 지도 중심 경험으로 방향을 바꾼 것
- 후보를 시/구 단위가 아니라 동네/상권 단위로 제한한 것
- AI를 후보 다양화 용도로 명확히 정의한 것
- 집앞 모드를 별도 재미 요소로 분리한 것
- 후보 상세, 주변 정보, 옵션을 접을 수 있게 정리한 것
- 방장 권한과 다시뽑기 동의 구조를 고민한 것
- 이름을 Drop으로 짧게 바꾼 것

## 7. 아쉬웠던 점

- 처음부터 디자인 시스템을 고정하지 않아 화면 톤이 여러 번 흔들렸다.
- 지도, DB, AI, 실시간 방을 한 번에 붙이면서 오류 범위가 커졌다.
- 후보 선정 기준이 초반에 명확하지 않아 "왜 이 장소가 나왔는지" 납득이 약했다.
- API별 역할 구분이 늦었다. 네이버 지도, 네이버 검색, ODsay, Directions 5의 책임을 더 빨리 나눴어야 했다.
- 실시간 방 기능은 상태 구조를 먼저 설계하고 들어갔어야 했다.
- 랜덤 연출은 처음부터 "사용자가 직접 선택한다"는 기준이 있었으면 덜 돌아갔을 것이다.

## 8. 앞으로의 원칙

### 제품 원칙

- 화면에는 지금 해야 할 행동만 남긴다.
- 후보는 넓은 지역명이 아니라 실제 만날 수 있는 동네/상권명으로 제공한다.
- 랜덤은 완전 무작위가 아니라 납득 가능한 범위 안에서만 작동한다.
- 집앞 모드는 장난스럽지만 공정해야 한다. 각 참여자별 극단 후보를 균등하게 준다.
- 설명보다 상호작용으로 재미를 만든다.

### 기술 원칙

- 지도 상태와 React 상태를 무리하게 묶지 않는다.
- AI 결과는 바로 쓰지 않고 검색 API로 검증한다.
- 가까운 참여자끼리는 검색 반경을 자동으로 줄인다.
- 후보가 부족하면 억지로 채우지 않는다.
- 실시간 방 기능은 최소 데이터만 구독한다.
- API 장애 시 fallback은 하되, 사용자에게는 조용히 자연스럽게 처리한다.

### UX 원칙

- 사람 추가 → 옵션 선택 → 후보 확인 → 랜덤 추첨 → 결과 확인 순서로 간다.
- 옵션은 기본값을 두고, 필요할 때만 펼친다.
- 후보 목록은 한눈에 삭제/확인 가능해야 한다.
- 주변 정보는 후보 상세 또는 결과 화면에서 보여준다.
- 버튼 이름은 짧고 직관적으로 쓴다.

## 9. 다음 단계

가장 먼저 안정화해야 할 부분은 세 가지다.

1. 가까운 참여자끼리의 후보 추천 정확도 개선
2. 대중교통/자차 소요시간 기반 후보 점수화
3. 공유방 실시간 동작 안정화

그 다음에는 Drop의 차별점인 랜덤 추첨 경험을 더 밀어야 한다.

- 카드 뽑기
- 컵 야바위
- 지도 다트
- 핀 드롭
- 지하철 도착 추첨

이 연출들은 단순 장식이 아니라 사용자가 "내가 직접 골랐다"고 느끼게 만드는 장치다.

## 10. 한 줄 회고

Drop은 처음에는 약속 장소를 공평하게 정하는 앱으로 시작했지만, 개발 과정에서 "친구들과 장소를 정하는 순간 자체를 재미있게 만드는 앱"으로 진화했다.  
가장 큰 배움은 좋은 랜덤은 아무 데나 던지는 것이 아니라, 모두가 납득할 수 있는 범위 안에서 의외성을 주는 것이라는 점이다.
